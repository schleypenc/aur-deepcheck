# AUR Supply-Chain Scanner — June 2026

Detection toolkit for the **atomic-lockfile / js-digest** AUR supply-chain incident.

This project combines:

1. **Community scanner** (`lenucksi/aur-malware-check`)
2. **Deep scanner** (`aur-deepcheck.sh`)

The goal is to detect both known compromised packages and post-compromise indicators that are not covered by the community scanner alone.

---

# Files

```text
aur-check.sh
aur-deepcheck.sh
README.md
```

`aur-check.sh` is the main entry point.

It automatically:

- clones or updates `lenucksi/aur-malware-check`
- executes the community scanner
- executes the deep pass
- returns a consolidated verdict

---

## Optional Dependency

`bpftool` is used for kernel BPF object enumeration and rootkit detection.

```bash
pacman -S bpf
```

Without it, section F (Kernel BPF Enumeration) will be skipped.

# Pass 1 — Community Scanner

Source: https://github.com/lenucksi/aur-malware-check

Checks:

- known compromised package names against pacman database and logs
- npm and bun caches for malicious packages
- known malware indicators
- predefined bpffs locations

## Package List Merging

Before pass 1 runs, `aur-check.sh` automatically merges two package lists:

| Source | URL | Coverage |
|--------|-----|----------|
| `lenucksi/aur-malware-check` | https://github.com/lenucksi/aur-malware-check | ~588 packages (curated) |
| AUR git mirror grep | https://gr.ht/aur_pkg_list.txt | 1500+ packages (generated directly from AUR git mirror) |

The merged and deduplicated list is used for the scan. The final count is printed at runtime:

```
[INFO]     package list: lenucksi(588) + gr.ht(1423) = 1847 unique packages
```

The `gr.ht` list is generated by grepping the AUR git mirror for commits containing
`atomic-lockfile` or `bun add` in the last 7 days. It is the most complete
list available and is fetched fresh on every run.

To disable the remote fetch and use the lenucksi list only:

```bash
sudo GRHT_URL='' ./aur-check.sh
```

# Why Clean Results Matter

A clean result means:

- no malicious package traces
- no cached malicious build artifacts
- no compromised pacman scriptlets
- no active payload process
- no visible rootkit indicators
- no suspicious persistence mechanisms

While no userland scanner can provide absolute guarantees, a fully clean run provides strong evidence that the malicious payload never successfully executed.

---

# Limitations

No userland tool can fully exclude:

- kernel modules
- advanced syscall hooking
- LSM abuse
- firmware persistence
- offline compromise

For high-assurance investigations, use trusted-media offline forensics.

---

# Incident Response

If CRITICAL findings are reported:

1. Disconnect the host immediately.
2. Rotate all credentials from a clean machine.
3. Review SSH keys and authorized_keys.
4. Preserve forensic evidence.
5. Reinstall from trusted media.
